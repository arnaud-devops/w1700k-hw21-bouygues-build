This audit covers the first custom v2 candidate based on Gilly's 19.07 source
and patch baseline. The target deployment is a W1700K HW2.1 on Bouygues DHCP,
WAN VLAN 100, routed DHCPv6-PD `/60`, no NAT6 and full dual-stack PPE offload.

The seven published files are the sysupgrade ITB, manifest, CycloneDX SBOM,
`sha256sums`, `profiles.json`, `config.diff` and `build-info.txt`. No backup,
raw capture or tar archive was published.

On the default case-insensitive macOS filesystem, standard iptables files such
as `xt_DSCP.ko` and `xt_dscp.ko` collide during rootfs extraction. The local
auditor uses `unsquashfs -force` for that host-only extraction step. The
case-distinct files remain intact in the SquashFS image and the Linux CI
extraction passed normally.

## Second-pass findings

### RTL8261CE match is not maximally strict

The bundled driver is byte-identical to Gilly 19.07. Its custom
`rtl8261ce_match_phy_device()` correctly requires Clause 45 and PMA/PMD model
`0x9`, thereby excluding RTL8261N model `0x2f`. It does not, however, call
`genphy_match_phy_device(phydev, phydrv)` before checking that model.

This matters because phylib uses a driver's custom `match_phy_device`
callback instead of its normal PHY-ID match. The current callback can
therefore theoretically claim an unrelated Clause-45 PHY that happens to
report PMA model `0x9`, without first proving the configured Realtek
vendor/model ID. This is not a known problem on the target HW2.1, whose PHY is
the expected RTL8261CE, but it is weaker than the requested strict matcher.

The follow-up candidate should restore the v1 guard:

```c
if (!phydev->is_c45 || !genphy_match_phy_device(phydev, phydrv))
	return 0;
```

and retain the existing PMA model check. The relevant phylib dispatch is in
[Linux v6.18 `phy_device.c`](https://github.com/torvalds/linux/blob/v6.18/drivers/net/phy/phy_device.c).

### Wi-Fi recovery readiness check is one revision behind

The embedded recovery command has the intended `flock` serialization and all
DoQ restoration settings. Its Wi-Fi readiness function counts operational
`/sys/class/net/*-ap*` interfaces, but it does not query the real
`hostapd.*` ubus objects and require `status=ENABLED` as the latest maintained
workspace version does.

This does not alter normal Wi-Fi startup and should work with the usual
`phy*-ap*` OpenWrt netdev names. It can nevertheless report recovery success
too early when an AP netdev is up while hostapd is not actually enabled. The
follow-up image should merge the hostapd-aware check while keeping `flock`.

### Optional FlowSense background probe

`npu-jitter` is enabled at boot and sends one ICMP probe to `1.1.1.1` every
two seconds to populate the FlowSense latency display. The traffic volume is
negligible and this is not a performance blocker, but it is a permanent
external probe. Keeping it enabled should be an explicit product choice; a
cleaner default would leave it disabled unless the live metric is wanted.

## Remaining hardware gate

This audit proves reproducibility and image contents; it does not prove runtime
behavior. The image has not been flashed. The active router remains on the
validated Gilly 17.07 image.

The recommended sequence is to fix the two specification gaps, rebuild and
repeat this offline audit before any hardware test. An explicitly authorized
hardware test will then still be required for boot and sysupgrade recovery,
both RTL8261CE links, WAN 10G, LAN2 multigig, sustained IPv4/IPv6 PPE traffic,
all three APs, DoQ, Watchcat, pstore and the irqbalance comparison. Until then,
neither this prerelease nor its follow-up may be treated as stable.
