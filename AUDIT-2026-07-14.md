# W1700K HW2.1 build audit - 2026-07-14

This audit targets one concrete deployment: W1700K HW2.1, Bouygues DHCP on
WAN VLAN 100, routed DHCPv6-PD /60, no NAT6, 10G WAN, multigig LAN and
three-band MT7996 Wi-Fi.

## Selected baseline

- OpenWrt/Fanboy UBI2: `7392ce326dd6f3299dc9ddf481a4f71c759a1e61`
- OpenWrt base under that branch: `b496e07d0f529804a4c4607284e263c6966a5b4b`
- Linux: `6.18.38`
- mt76: `59676919ea408b0b13a9d23f2e2e1a1ab407fba1`
- hostapd/wpa_supplicant: stable `2_12` branch at
  `f08f2749aa696c4e47c5c0f591dda99951bf9fac`

Fanboy rebased UBI2 after the first candidate. The only source-tree change
between the old pin `271d907218` and the selected pin is OpenWrt's U-Boot
2026.07 update. It does not change the running kernel, rootfs, W1700K driver
or flow-offload path in a sysupgrade image. The new pin is nevertheless used
so a fresh build starts from the current UBI2 branch state.

Linux 6.18.38 is the latest 6.18 stable release available on the audit date.
OpenWrt main also uses 6.18.38. The mt76 pin is the current OpenWrt mt76
master. The hostapd pin is the current stable 2.12 branch and is the version
from OpenWrt PR 24142, which has W1700K runtime reports and approvals.

## Compared public images

The newest public images found during the final check were Fanboy's non-OC
`ubi2_2026.07.13_r35355-271d907218`, its separate overclocked variant, and
Gilly's `14.07.2026_universal` image (`r0+35335-b496e07d0f`). No newer
non-overclocked UBI2 or Gilly image was published before this audit closed.

Gilly 14.07 is a useful generic test image: it has Linux 6.18.38, the
selective IPv6/VLAN patch, the hardened Wi-Fi flowtable hook and full
`wpad-mbedtls`. It still carries the April 2 hostapd snapshot and does not
embed dnsproxy or this deployment's recovery policy. The custom profile keeps
the proven Gilly fixes while using the July 9 hostapd snapshot,
`wpad-openssl`, built-in dnsproxy and pinned deployment files.

The Fanboy overclocked image is deliberately not used as a baseline. The
stock 1200 MHz image already reaches the client and ISP limits through PPE,
while overclocking adds thermal and stability risk without fixing the IPv6
VLAN limitation.

## Driver and firmware decisions

The build retains Fanboy's dedicated RTL8261CE driver and the normal
`rtl8261n.bin` firmware. This combination is already validated on HW2.1:

- WAN binds as RTL8261CE and negotiates 10 Gbit/s with the ONT;
- LAN2 binds as RTL8261CE and negotiates 2.5 Gbit/s with the available client;
- no Generic Clause 45 fallback or firmware-load error was observed.

Gilly's universal tree contains one newer driver-matching guard that is absent
from Fanboy: it reads the PMA/PMD model before allowing the dedicated CE driver
to bind. The build carries that guard with an additional
`genphy_match_phy_device()` check, because a custom phylib matcher replaces the
normal vendor-ID comparison. This is behavior-neutral for genuine RTL8261CE
silicon and prevents the CE driver from claiming an RTL8261N or an unrelated
Clause 45 PHY if both Realtek drivers are ever present.

OpenWrt PR 23644 is the preferred future upstream-shaped RTL8261C/D/CE
driver, but it remains open and blocked. It has positive LAN2 and RTL8261D
reports, while final HW2.1 WAN validation is still incomplete. Mixing its
new firmware with the current vendor-derived driver is not valid; driver and
firmware must move together after the PR is accepted and tested on this WAN.

The selected MT7996 WM, WA, DSP and ROM patch blobs and both EN7581 MT7996
NPU blobs are byte-for-byte identical to current linux-firmware main on the
audit date. The RTL8261 payload arrays are unchanged in the newer upstream
firmware repository commits; those commits contain build-endianness,
licensing and repository cleanup. The normal firmware is retained instead of
the low-power variant to avoid an unneeded PHY behavior change.

The selected Fanboy tree also already carries the MT7996 firmware SKU/backoff
power-table controls (`0010`) and their per-radio refresh path (`0011`) used by
Gilly's latest image. They are therefore not lost when moving away from the
Gilly test image, and no duplicate local patch is needed.

## Included local fixes

Every source overlay is covered by `source-files.sha256`; a missing, added or
changed file aborts the workflow.

| Area | Fix | Reason |
| --- | --- | --- |
| PPE | selective IPv6/VLAN uplink fallback (`965`) | Hardware-tested fix for the routed IPv6 VLAN 100 black-hole; IPv4 and safe IPv6 directions remain eligible for PPE. |
| PPE | PPE1 queue reservation (`948`) | Accepted upstream as Linux commit `c66f8511a810`. |
| PPE | SRAM flush error propagation (`949`) | Accepted upstream as Linux commit `d7d81b003013`. |
| PPE | RCU grace period before NPU release (`968`) | Prevents teardown from releasing an NPU object still visible to an RCU reader. |
| Thermal | sensor interval mask (`960`) | Fixes an obvious wrong-field write; also carried by the reviewed AN7583 thermal series. |
| Thermal | low-trip clamp (`961`) | Fixes an obvious wrong-variable clamp; also carried by the reviewed AN7583 thermal series. |
| GRO | reject flush-marked skb list aggregation (`992`) | Small v4 net fix preventing invalid double aggregation in a path relevant to HW GRO. |
| MT7996 | three HE/EHT NULL guards (`016`-`018`) | Backports Lorenzo Bianconi's defensive mt76 series. |
| MT7996 | truncated NPU tx_free guard (`019`) | Keeps already parsed entries releasable when the AN7581 descriptor count reaches the buffer edge. |
| RTL8261CE | exact CE model matcher (`969`) | Retains phylib's vendor match and prevents the dedicated CE driver from claiming sibling RTL826x silicon. |
| Wi-Fi/PPE | serialized AP hotplug reload | Waits for concurrent fw4 reloads so the last-created AP is not omitted from the flowtable. |

The active Gilly 13.07 image already runs the kernel and mt76 safety fixes
above without an observed warning or regression. The serialized hotplug
variant is taken from Gilly's 14.07 hardening.

## Deliberately excluded

- OpenWrt PR 24038 native bridge flow offload: architecturally preferable to
  `nf_call_iptables`, but still a blocked draft with a reported W1700K
  reconnect regression. It should be tested as a separate experimental image,
  not mixed into the production candidate.
- OpenWrt PR 22533 refreshes: the selected UBI2 tree already contains the
  proven bridge-offload stack used by the current router.
- TRTCM retry patch `946`: its v2 review identified unresolved state-machine
  and retry semantics. This deployment does not use hardware HTB shaping.
- ETS patch `962`, eager pure-bridge PPE initialization, and the large MIB
  rewrite: not needed for this routed profile and not justified by its tested
  workload.
- mt7996 power-save revert `015`: the current upstream behavior is retained
  because the router has no matching stall and the revert is a local
  workaround.
- CAKE/SQM: shaping at an 8 Gbit/s access rate would bypass the intended PPE
  fast path and cannot be treated as a free bufferbloat improvement.
- CPU overclock: stock 1200 MHz with `ondemand` already sustains the tested
  multigig offloaded path; thermal and stability margin are more valuable.

## Package policy

The image embeds `wpad-openssl`, `dnsproxy`, `dnsmasq-full`, `ethtool-full`,
the RTL8261CE module/firmware and MT7996/NPU firmware. `wpad-basic-*` remains
forbidden. Full ethtool is retained for 10G/2.5G PHY negotiation and link-mode
diagnostics; unrelated WireGuard, CAKE, TPROXY and ext4 utilities from the
generic Gilly image are not imported into this routed profile.

`owut` and LuCI Attended Sysupgrade are excluded. Their generic build service
cannot reproduce this pinned source and local patchset, so presenting them as
a supported upgrade path risks silently losing the selective fix, full wpad
or dnsproxy. Upgrades remain checksum-verified local sysupgrades through the
guarded workspace script.

The inherited `github_check` and `github_fetch` CGI helpers are removed as
well. They download generic `w1700k/builds` firmware rather than this custom
profile and were reachable even when the LuCI upgrade client was absent. The
single-wiphy LuCI channel-analysis patch is retained independently.

`CONFIG_ALL_KMODS` is removed because it only compiles every unrelated kernel
package and expands build time/cache. Device-required modules remain selected
by the W1700K image profile and are asserted again against the final manifest.

Fanboy's generic builder normally injects the current official OpenWrt
buildbot `vermagic` into locally patched kernels and exposes the matching
official `kmods` feed. This profile deliberately does neither. A module built
against the unpatched buildbot kernel must not be presented as ABI-compatible
with this image merely because both report Linux 6.18.38. Every module needed
by this W1700K is compiled and embedded in the image; adding another kernel
module requires rebuilding this profile with that module selected.

The remaining official APK feeds are userspace-only and their exact repository
list is versioned with the profile. All rootfs profile files, including that
list and the DoQ/Wi-Fi recovery files, are covered by
`profile-files.sha256`; the profile hook is covered separately by
`profile-hooks.sha256`. The build fails if locked content changes without an
explicit lock update.
The native-ABI policy has its own build-cache epoch and discards pre-existing
target staging once, preventing an old forced-vermagic module from entering a
new image through an incremental cache.

Once the stock dual-stack fw4 rule is detected after first boot, the UCI
default removes the package-owned `ruleset.uc` path from `/etc/sysupgrade.conf`.
This avoids carrying an old fw4 template into a later firewall4 package while
preserving the documented IPv4-only fallback procedure if it is ever needed.

Before release creation, CI inspects the assembled rootfs itself. It checks
the locked recovery files and executable modes, full wpad links, firmware and
driver modules, dual-stack fw4 rule, native-ABI feed policy, and the absence of
generic upgrade clients and CGI download helpers.

The recovery files remain listed in `/etc/sysupgrade.conf` so they survive a
future generic image that omits dnsproxy. On this custom profile, the first
boot script copies the current audited versions back from `/rom` before
enabling the service. This prevents a preserved older overlay copy from
masking a fix shipped in a newer image. Recovery serialization uses `flock`,
whose kernel-held lock is released even if the process terminates abruptly.

## Release gate

No image from this audit is considered stable until all of the following pass
on the actual HW2.1 board:

1. OpenWrt metadata and `sysupgrade -T` validation without `--force`.
2. RTL8261CE binding on WAN and LAN2, with WAN 10G negotiation.
3. 2.4, 5 and 6 GHz AP startup, DFS completion and reconnect testing.
4. AdGuard DoQ primary, NextDNS DoQ fallback and DNS.SB bootstrap recovery.
5. Normal IPv4+IPv6 fw4 flow rule with hardware offload enabled.
6. Sustained IPv4 and IPv6 WAN tests on 2.5G Ethernet and the 1G Pi.
7. PPE inspection confirming no IPv6 WAN-uplink BND entry that requires VLAN
   100 insertion, while IPv4 VLAN offload remains active.
8. Kernel log inspection for NPU, mt7996, PHY, RCU, GRO and thermal errors.

Building and publishing does not authorize flashing the router.
